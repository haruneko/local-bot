# embed モデル選定の証拠（2026-06-17）

> 状態: cited（CLAUDE.md §embed が出典として引く）／2026-06-17 凍結。
> 記事版（経緯の読み物）は `article/embedding-locate-eval-2026-06-17.md`。ここは決定事実のみ。

## 決定

embedModel = **ruri-v3-310m**（768 次元・日本語特化・query/doc 非対称接頭辞）。旧 `nomic-embed-text` から差し替え。

## 根拠（自前データ実測 `npm run eval:retrieval`）

公開ベンチ（JMTEB 等）は候補の絞り込みまで。最終決定は自前 gold で測った。

### memo locate（gold 40 件・MRR）

| kind | nomic | bge-m3 | ruri-v3 |
|---|---|---|---|
| ALL | 0.78 | 0.95 | 0.93 |
| 名前直打ち(英名) | 1.00 | 1.00 | 0.87 |
| 言い換え(topical) | 0.79 | 0.98 | 0.98 |
| 暗示(oblique) | 0.48 | 0.81 | 0.76 |
| holdout(伏せ) | 0.44 | 0.87 | 1.00 |

nomic は接頭辞を付けても 0.78 止まり、本番で最多の「暗示」が 0.48 で壊滅。holdout でも bge/ruri が圧勝＝過学習ではない。

### エピソード想起（733 件・top-k 10）

| | R@1 | R@10 | MRR |
|---|---|---|---|
| nomic | 7% | 27% | 0.15 |
| bge-m3 | 50% | 67% | 0.58 |
| ruri-v3 | 40% | 83% | 0.56 |

nomic は想起全体が腐っていた。bge-m3 ≈ ruri-v3。

### bge と ruri は本文だけ見れば互角

bge の優位は「英語ファイル名を意味ベクトルに混ぜていた」産物。ファイル名を埋め込みから外すと name 行は両者 0.82 で並び、全体は ruri 0.93 = bge 0.92。一般ベンチの日本語 SOTA という評判と一致。

### 速度（ウォームアップ後・中央値/1回）

nomic 38ms / **ruri-v3 85ms** / bge-m3 209ms。ruri は bge の約 2.4 倍速い。毎ターン embed＋reindex 全件回しなので速度が本丸。

## 効いている設計上の事実

- 品質互角 → 運用要因（速度・次元・統合コスト）で決める。ruri は 768 次元で nomic と同次元＝移行が楽。
- 代償＝query/doc 接頭辞のコード配線（`src/llm/embed-prefix.ts`・ruri は非対称 `検索クエリ:`/`検索文書:`）。テストで担保。
- 評価の前提＝コーパスの清潔さ。テストが本番 notes に書き込んでコーパス汚染するバグを先に潰した。
- 識別子（ファイル名）を意味ベクトルに混ぜてよいかは未解決の設計問題として残る。
