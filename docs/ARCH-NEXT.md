## 0. 何を目指して、何をやらないか

**目指すこと**: 体があって、そこに居る感じ。連続した時間の中で、見聞きしながら、いろんな速さで反応する存在。

**やらないこと（破ると embodied-claude の劣化コピーになる）**:

1. **欲を「○○したい」と言葉にして持たせない。** 願いを設定として書かない。行動は、気分（affect）・気にかけてること（concern）・見聞きしたもの・各モジュールが自分で動くか決めること、から**結果として**出る。「見たかったんだね」は動いた後に内省が振り返って言うだけ。
2. **感情を数値やラベルにしない。** affect は「生の気分の言葉」のまま。点数や感情ベクトルを足さない。
3. **でかいモデル1個に全部やらせない。** omni（後述）は目・耳・口だけ。考える・決めるは別のモジュールに分ける。omni に全部飲ませると「omni が一人で決める」＝捨てたはずの偉い決定役が戻る。
4. **見聞きしたものを文章に起こして「これが事実」と保存しない。** 見聞きは「動きのきっかけ」であって、書き留めた言葉ではない。

## 1. 全体の形

### 1.1 板が1枚（共有 Context）

- モジュールは**板にだけ書き、板からだけ読む**。モジュール同士は直接喋らない（必ず板を介す）。
- **各モジュールは、自分に要るチャンネルだけ**板から受け取る（今の `actors[name].channels` の宣言を拡張するだけ。生の画像/音声チャンネルを足し、それを宣言したモジュールだけが受け取る）。
- 板への書き込みと、ファイル/DB への確定だけがアトミック（つまり、途中半端な状態を他から見せない）。

### 1.2 全部が「自分で動くか決める」モジュール

- 今の actor だけでなく、**言語野・内省・内心更新（affect/concern/importance）も全部 actor にする**。板の上で全員が同じ作法（自分で動くか決める・黙ってもいい）で並ぶ。
- 「見たものを言葉にできるか」は全体で決めず、**内省くんが omni チャンネル（生の画像/音声）を取るかどうか**で決まる。語らせたければ渡す、いらなければ渡さない＝モジュール単位の選択。
- 順番が要る所（発話・行動 → 内省 → affect/importance）は、**各モジュールの要るチャンネルが板に乗るまで動かない**ことで自然に出る。固定の順番表は要らない。

### 1.3 固定のフェーズ順を捨てて、フレームにする

- 今は SPEC §4.2 でフェーズの順番を固定してる。これをやめる。
- 代わりに**フレーム**＝「時計でカチッと進む、文脈のコマ」（パラパラ漫画の1枚）。人の文脈が毎瞬間ゆるやかに切り替わるのに近い。
- フレームは**時計で進む**（誰かの書き込みを待たない）。だから「待ち＝順番＝ターンの作り直し」にならない。
- 注意: 「依存が揃ったら動く」を追記だけでやると、結局みんな前の書き込みを待つ＝**隠れたターン制**になる。それを避けるのがフレーム（時計駆動）。

### 1.4 ターン版と連続版は同じコード・時計が違うだけ

- **順番版** = フレームを「今のモジュールが終わったら1個進める」（時計＝処理完了）。決定的でテストしやすい。今すぐ動く。
- **連続版** = フレームを「実時間で進める」（時計＝壁掛け時計）。これが目指す形。
- 両者は択一ではなく**時計の設定違い**。だから順番版は捨てない——連続版の時計を「完了駆動」に差し替えただけ。実験的すぎる連続版に賭けて、使える順番版を捨てる愚を避ける。

### 1.5 フレームレートはモジュールごと

- 1個の細かい時計を、各モジュールが**自分の速さで間引いて読む**。とっさの返事は毎フレーム、考えた発話は数フレームに1回。
- ＝「速さが違うモジュール」が「サンプリング速度が違う」として表れる。

### 1.6 起動の段（none / systematic / llm）— モジュールごとに設定

「動くか決める」判断には強さの段があり、モジュールごとに設定で選ぶ。全部を LLM に判断させない。

- **none（常に動く）**: 判断するだけ無駄なもの。例＝想起（recall）。走らせる。
- **systematic（機械的ゲート）**: データ条件で決める・LLM を使わない。例＝視覚の見分けは「`image_feed` に画像が乗っている時だけ動く」。判断ではなくただの if。
- **llm（軽い activator）**: 問いの性質を読む必要があるもの。例＝webSearch（外を調べる要る？）。小型モデルで yes/no ＋一言。

これで **preprocess と actor の線引きが消える**＝どちらも同じモジュールで、違いは起動の段だけ（recall=none / 視覚見分け=systematic / webSearch=llm）。
フレームが速く回っても cheap なモジュールは none/systematic で LLM を呼ばないので、**§4 で棚上げした「活性化コスト」はこの設計で解決**する。
設定は `actors[name].enabled` の延長で `activate: none | systematic | llm` を持つ形。

## 2. 板に乗る主な部品

### 2.1 目と耳と口（omni・文字起こししない）

- **Qwen3-Omni**（30B-A3B＝実際に動くのは約3B・EVO-X2 に乗る大きさ）を、言語野が画像/音声も扱える部品として使う。画像/音声を**そのまま**入れて**そのまま音声で**返せる＝文字起こしも読み上げも挟まない（2026 時点でローカル可）。
- omni は目耳口だけ（§0-3）。生の画像/音声は板の専用チャンネルに乗り、それを取るモジュールだけが読む。
- 記憶への残し方: ベクトル（embedding）＋そのときの気分（affect）。説明文を正本にしない。

### 2.2 とっさの返事（即時返答）

- 相手が喋った瞬間の短い一言（「ん」「おかえり」… または無）。考える前に口が動くやつ。自分で出すか決める・黙ってもいい。一番軽くて速いモデル・最小の材料。
- 板に乗るので、考えた発話（遅い）は**板の上のとっさの一言を見て**、続ける／言い直す／黙る、を選べる。
- とっさの一言は**作業記憶に乗せて消す**（長期記憶には残さない方向）。人が「とりあえずの返事を作業記憶に乗せて消す」のに倣う。

### 2.3 別々の思い出し（速さの違う流れ）

- 思い出し（recall）を1回まとめてではなく、**各モジュールが自分のタイミングで**やる。とっさの返事＝ほぼ思い出さない一番速い流れ、actor＋熟考＝中、じっくり思い出し/調べ＝遅い。
- できた順に板へ。相手に届く言葉は時間に沿って混ざって出る。
- 遅い流れは、出来た頃には文脈が動いてる（古い板に答える）。これは**反応の遅れとして許容**し、ダメならフレームレートで調整する（読み直しの要否は §4）。

## 知覚と記憶ループ（画像・音声をどう覚えるか）

知覚（画像・今後は音声）を、内省・メモ・夢の各ループがどう扱うか。原則: **キャプション（説明文）を正本にしない**・**モダリティに依らず同じ形**。

### 人間の視覚記憶に倣う

人はピクセルを貯めていない。**圧縮した非言語の痕跡（embedding）＋意味のラベル**を持ち、思い出す時に作り直す（再構成・歪む）。顔は「見れば分かる（recognition）が描けない（recall）」＝embedding を持つから見分けられる。だから**視覚記憶の核はピクセルではなく embedding**。

### 記憶のしかた（3層・役割が違う）

- **言葉（意味ラベル）**: 内省が「会話で起きたこと＝エバの反応」をエピソードに書く（既存・v1）。画像のキャプションではなく彼女の言葉。
- **embedding（視覚記憶の核）**: 画像を embed したベクトルをエピソードに一緒に保存。新しい画像が来たら embed して**似た過去を想起（recognition）**。非言語のまま・キャプションしない。符号化時に embed してピクセルは捨てる＝人間と同じ。想起は再構成的＝ふんわり想起（full/vague）に乗る。**画像用の埋め込みモデル（CLIP 系）が要る**（サーバに今は無い・要 pull）。
- **ピクセル（写真アルバム・脳には無い超人機能）**: 残すのは**能動メモ限定**。エバが「とっておこう」と決めた画像だけ `data/notes/` に保存しノートから参照。受動の記憶＝揮発／能動の記録＝残す、の B'（記憶 vs 記録）に乗る。受動のエピソードはピクセルを持たない。

### 各ループの扱い

- **内省**: テキストのまま。画像は見ない（見るとキャプション化に逆戻り）。ピクセルを残す時も内省は**参照（ポインタ）だけ**付ける。**見直すのは想起の時で符号化の時じゃない**＝内省はずっと multimodal でなくていい。
- **能動メモ**: ピクセルを保存する唯一の経路（上記）。binary をノートに書くプリミティブが要る。
- **夢（蒸留）**: テキストのまま。画像は再蒸留しない（メモ本文は要約しない硬いルールと同じ）。視覚のパターンは彼女が繰り返し言及すればテキスト経由で勝手に蒸留される。

### 音声も同じ形

声・音の embedding で「あの声だ」を見分ける。器（embedding が核・波形は能動メモ・内省と夢はテキスト）はモダリティに依らず共通。`ChatMessage.images` の隣に後で `audio` が並ぶ。

## embedding サブシステム（モダリティごとに backend を差し替え）

`LlmClient`（chat を差し替え可能にしてる）と同じ思想で、embedding も backend を差し替え可能にする。

- **窓口は TS**: `EmbeddingProvider`（`embedText` / `embedImage`）。呼ぶ側（recall・視覚想起）は**どこで embed してるか知らない**。
- **backend はモダリティごと**:
  - text → Ollama `nomic-embed-text`（既存・HTTP）
  - image → 別 backend（下記）。落ちてる/未設定なら **null を返す**＝視覚見分けは systematic 起動なのでただスキップ（壊れない・Slack の files:read 無しと同じ degrade）。
- **2026-06-15 に判明: Ollama は画像 embed 不可**。専用 embed モデルは `images` を無視、vision 生成モデルは `/api/embed` が **501 Not Implemented**。上流 issue（multimodal embedding 対応）は open。＝画像 embedding は Ollama の外で回すしかない。
- **画像 backend の選択肢**:
  - **in-process（推し）**: transformers.js / onnxruntime-node で CLIP を Node プロセス内で回す。外部サービスも subprocess も無し＝ただのライブラリ呼び出し。初回に ONNX を DL してキャッシュ。
  - 外部 HTTP サーバ（Python・nomic-embed-vision を常駐）: 別プロセス管理が要る（ops がめんどい）。
- **クロスモーダル（言葉↔画像を同空間で）＝保留だが捨てない**: nomic-embed-text ↔ nomic-embed-vision は同空間だが Ollama が画像を出せない。in-process で同空間にするなら text も画像も jina-clip 等に寄せる＝**既存エピソードの再 embed（移行）が要る**。recognition（画像→画像）は別空間でも効くので、先に画像 backend だけでも視覚記憶は立つ。クロスモーダルはその後の判断。
- **将来**: Ollama が画像 embed 対応したら image backend を Ollama に差し替えるだけ（窓口・呼ぶ側は無変更）。

## 4. あとで決めること（設計が固まってから・実装時のツマミ）

設計に混ぜない。懸念としては正しいので記録だけする。

- **フレームレートの値**（1個の細かい時計の速さ、各モジュールの間引き幅）。
- **遅いモジュールの読み直し**: 出来た時に過ぎたフレームを読み直して「まだ言う?」を判断するか。毎回やる（安全・重い）か、変化が一定以上の時だけ（軽い・「一定以上」を何で測るか要定義）か。今は「遅れ許容＋フレームレート調整」で先送り。
- **フレームの寿命**: 板にどれだけ残すか（古いフレームの落とし方）。長期に残すものは内省が LanceDB に符号化。
- ~~**活性化のコスト**~~ → §1.6「起動の段（none / systematic / llm）」で解決。cheap なモジュールは none/systematic で LLM を呼ばない。残るのは llm 段のモジュールを「入力が変わった時だけ再判断」するか等の実装ツマミだけ。
- **とっさの一言を内省がエピソードに含めるか**（作業記憶＝乗せる／長期＝残さない、までは合意。境界の明文化は実装時）。
- `LlmClient`/Ollama の streaming 対応と omni の運用（VRAM・`ollamaMaxConcurrency` の同時実行制限との兼ね合い）。
