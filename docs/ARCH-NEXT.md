## 0. 何を目指して、何をやらないか

**目指すこと**: 体があって、そこに居る感じ。連続した時間の中で、見聞きしながら、いろんな速さで反応する存在。

**やらないこと（破ると embodied-claude の劣化コピーになる）**:

1. **欲を「○○したい」と言葉にして持たせない。** 願いを設定として書かない。行動は、気分（affect）・気にかけてること（concern）・見聞きしたもの・各モジュールが自分で動くか決めること、から**結果として**出る。「見たかったんだね」は動いた後に内省が振り返って言うだけ。
2. **感情を数値やラベルにしない。** affect は「生の気分の言葉」のまま。点数や感情ベクトルを足さない。
3. **でかいモデル1個に全部やらせない。** omni（後述）は目・耳・口だけ。考える・決めるは別のモジュールに分ける。omni に全部飲ませると「omni が一人で決める」＝捨てたはずの偉い決定役が戻る。
4. **見聞きしたものを文章に起こして「これが事実」と保存しない。** 見聞きは「動きのきっかけ」であって、書き留めた言葉ではない。

## 1. 全体の形

### 1.1 板が1枚（共有 Context）

- モジュールは**板にだけ書き、板からだけ読む**。モジュール同士は直接喋らない（必ず板を介す）。
- **各モジュールは、自分に要るチャンネルだけ**板から受け取る（今の `actors[name].channels` の宣言を拡張するだけ。生の画像/音声チャンネルを足し、それを宣言したモジュールだけが受け取る）。
- 板への書き込みと、ファイル/DB への確定だけがアトミック（つまり、途中半端な状態を他から見せない）。

### 1.2 全部が「自分で動くか決める」モジュール

- 今の actor だけでなく、**言語野・内省・内心更新（affect/concern/importance）も全部 actor にする**。板の上で全員が同じ作法（自分で動くか決める・黙ってもいい）で並ぶ。
- 「見たものを言葉にできるか」は全体で決めず、**内省くんが omni チャンネル（生の画像/音声）を取るかどうか**で決まる。語らせたければ渡す、いらなければ渡さない＝モジュール単位の選択。
- 順番が要る所（発話・行動 → 内省 → affect/importance）は、**各モジュールの要るチャンネルが板に乗るまで動かない**ことで自然に出る。固定の順番表は要らない。

### 1.3 固定のフェーズ順を捨てて、フレームにする

- 今は SPEC §4.2 でフェーズの順番を固定してる。これをやめる。
- 代わりに**フレーム**＝「時計でカチッと進む、文脈のコマ」（パラパラ漫画の1枚）。人の文脈が毎瞬間ゆるやかに切り替わるのに近い。
- フレームは**時計で進む**（誰かの書き込みを待たない）。だから「待ち＝順番＝ターンの作り直し」にならない。
- 注意: 「依存が揃ったら動く」を追記だけでやると、結局みんな前の書き込みを待つ＝**隠れたターン制**になる。それを避けるのがフレーム（時計駆動）。

### 1.4 ターン版と連続版は同じコード・時計が違うだけ

- **順番版** = フレームを「今のモジュールが終わったら1個進める」（時計＝処理完了）。決定的でテストしやすい。今すぐ動く。
- **連続版** = フレームを「実時間で進める」（時計＝壁掛け時計）。これが目指す形。
- 両者は択一ではなく**時計の設定違い**。だから順番版は捨てない——連続版の時計を「完了駆動」に差し替えただけ。実験的すぎる連続版に賭けて、使える順番版を捨てる愚を避ける。

### 1.5 フレームレートはモジュールごと

- 1個の細かい時計を、各モジュールが**自分の速さで間引いて読む**。とっさの返事は毎フレーム、考えた発話は数フレームに1回。
- ＝「速さが違うモジュール」が「サンプリング速度が違う」として表れる。

### 1.6 起動の段（none / systematic / llm）— モジュールごとに設定

「動くか決める」判断には強さの段があり、モジュールごとに設定で選ぶ。全部を LLM に判断させない。

- **none（常に動く）**: 判断するだけ無駄なもの。例＝想起（recall）。走らせる。
- **systematic（機械的ゲート）**: データ条件で決める・LLM を使わない。例＝視覚の見分けは「`image_feed` に画像が乗っている時だけ動く」。判断ではなくただの if。
- **llm（軽い activator）**: 問いの性質を読む必要があるもの。例＝webSearch（外を調べる要る？）。小型モデルで yes/no ＋一言。

これで **preprocess と actor の線引きが消える**＝どちらも同じモジュールで、違いは起動の段だけ（recall=none / 視覚見分け=systematic / webSearch=llm）。
フレームが速く回っても cheap なモジュールは none/systematic で LLM を呼ばないので、**§4 で棚上げした「活性化コスト」はこの設計で解決**する。
設定は `actors[name].enabled` の延長で `activate: none | systematic | llm` を持つ形。

## 2. 板に乗る主な部品

### 2.1 目と耳と口（omni・文字起こししない）

- **Qwen3-Omni**（30B-A3B＝実際に動くのは約3B・EVO-X2 に乗る大きさ）を、言語野が画像/音声も扱える部品として使う。画像/音声を**そのまま**入れて**そのまま音声で**返せる＝文字起こしも読み上げも挟まない（2026 時点でローカル可）。
- omni は目耳口だけ（§0-3）。生の画像/音声は板の専用チャンネルに乗り、それを取るモジュールだけが読む。
- 記憶への残し方: ベクトル（embedding）＋そのときの気分（affect）。説明文を正本にしない。

### 2.2 とっさの返事（即時返答）

- 相手が喋った瞬間の短い一言（「ん」「おかえり」… または無）。考える前に口が動くやつ。自分で出すか決める・黙ってもいい。一番軽くて速いモデル・最小の材料。
- 板に乗るので、考えた発話（遅い）は**板の上のとっさの一言を見て**、続ける／言い直す／黙る、を選べる。
- とっさの一言は**作業記憶に乗せて消す**（長期記憶には残さない方向）。人が「とりあえずの返事を作業記憶に乗せて消す」のに倣う。

### 2.3 別々の思い出し（速さの違う流れ）

- 思い出し（recall）を1回まとめてではなく、**各モジュールが自分のタイミングで**やる。とっさの返事＝ほぼ思い出さない一番速い流れ、actor＋熟考＝中、じっくり思い出し/調べ＝遅い。
- できた順に板へ。相手に届く言葉は時間に沿って混ざって出る。
- 遅い流れは、出来た頃には文脈が動いてる（古い板に答える）。これは**反応の遅れとして許容**し、ダメならフレームレートで調整する（読み直しの要否は §4）。

## 知覚と記憶ループ（画像・音声をどう覚えるか）

知覚（画像・今後は音声）を、内省・メモ・夢の各ループがどう扱うか。原則: **キャプション（説明文）を正本にしない**・**モダリティに依らず同じ形**。

### 人間の視覚記憶に倣う

人はピクセルを貯めていない。**圧縮した非言語の痕跡（embedding）＋意味のラベル**を持ち、思い出す時に作り直す（再構成・歪む）。顔は「見れば分かる（recognition）が描けない（recall）」＝embedding を持つから見分けられる。だから**視覚記憶の核はピクセルではなく embedding**。

### 記憶のしかた（3層・役割が違う）

- **言葉（意味ラベル）**: 内省が「会話で起きたこと＝エバの反応」をエピソードに書く（既存・v1）。画像のキャプションではなく彼女の言葉。
- **embedding（視覚記憶の核）**: 画像を embed したベクトルをエピソードに一緒に保存。新しい画像が来たら embed して**似た過去を想起（recognition）**。非言語のまま・キャプションしない。符号化時に embed してピクセルは捨てる＝人間と同じ。想起は再構成的（full/summarize、遠いものは omit）。embedding は **横断モデル（ImageBind・下のサブシステム）** に乗せる＝画像/音/文字が1空間（2026-06-16 に Docker/CPU で実測確定）。
- **ピクセル（写真アルバム・脳には無い超人機能）**: 残すのは**能動メモ限定**。エバが「とっておこう」と決めた画像だけ `data/notes/` に保存しノートから参照。受動の記憶＝揮発／能動の記録＝残す、の B'（記憶 vs 記録）に乗る。受動のエピソードはピクセルを持たない。

### 各ループの扱い

- **内省**: テキストのまま。画像は見ない（見るとキャプション化に逆戻り）。ピクセルを残す時も内省は**参照（ポインタ）だけ**付ける。**見直すのは想起の時で符号化の時じゃない**＝内省はずっと multimodal でなくていい。
- **能動メモ**: ピクセルを保存する唯一の経路（上記）。binary をノートに書くプリミティブが要る。
- **夢（蒸留）**: テキストのまま。画像は再蒸留しない（メモ本文は要約しない硬いルールと同じ）。視覚のパターンは彼女が繰り返し言及すればテキスト経由で勝手に蒸留される。

### 音声も同じ形

声・音の embedding で「あの声だ」を見分ける。器（embedding が核・波形は能動メモ・内省と夢はテキスト）はモダリティに依らず共通。`ChatMessage.images` の隣に後で `audio` が並ぶ。

## embedding サブシステム（モダリティごとに backend を差し替え）

`LlmClient`（chat を差し替え可能にしてる）と同じ思想で、embedding も backend を差し替え可能にする。

> **読み替え（2026-06-17）**: 本節の text embedder は **2026-06-17 に nomic → ruri-v3 に変更**（日本語特化・実測で想起が大幅改善）。**768 次元・`OllamaEmbedClient`・横断とは別空間という構造はそのまま**なので、以下に出る "nomic" は **ruri-v3** と読み替えること（モデル名だけの差・横断設計は不変）。

- **窓口は TS**: text は既存 `OllamaEmbedClient`（nomic）、横断は `XmodalEmbedder`（ImageBind・画像/音/文字を 1 空間）。呼ぶ側（recall・視覚/聴覚想起）は**どこで embed してるか知らない**（落ちてたら null＝degrade）。
- **デュアルベクトル（2026-06-16 確定）**: エピソードは**2本のベクトルを両持ち**する。① **text → `ruri-v3`（768・Ollama HTTP。2026-06-17 に nomic から変更）**＝text-text 想起の質を落とさない（メモ本文を要約しない硬ルールと同じ"劣化禁止"思想）。② **横断 → ImageBind（1024・下記）**＝画像/音/文字を**1つの共有空間**に縛り、言葉を介さない連想（声→顔・匂い→情景）を効かせる。recall は2空間を引いてマージ。
  - 完全統一（text も ImageBind のみ）は ImageBind の text encoder が純テキスト検索で弱く、既存エピソード全件の再 embed も要るので**却下**。ハイブリッド（text=nomic・知覚=ImageBind だが**別空間**）は横断が効かず**却下**。
  - **「既存エピソード再 embed 移行」はデュアルだと不要**＝既存の text-only はそのまま、ImageBind ベクトルは**これから入る知覚エピソード（画像/音）にだけ**付く。
- **横断 backend ＝ ImageBind on Docker（CPU・2026-06-16 実測で確定）**:
  - ImageBind（Meta・1.2B）を **Python の Docker HTTP 常駐サービス**にして bot は HTTP で叩く。ImageBind は PyTorch 物で in-process（transformers.js）に乗らないので、これが素直。落ちてる/未設定なら **null を返す**＝知覚見分けは systematic 起動なのでただスキップ（壊れない・Slack の files:read 無しと同じ degrade）。
  - 実測（Strix Halo・CPU 16スレッド・単発 forward median）: **text 137ms / audio 245ms(5s) / image 649ms**、共有 **dim=1024**、**GPU 不使用**（Ollama=iGPU と計算ユニット別＝並列・VRAM 非競合）。起動ロード 21s（常駐なら1回）。常駐メモリ fp32 ~4.8GB（統合プール共有・**重さがダメなら fp16~2.5/int8~1.2GB へ量子化**）。
  - 検証 scaffold は上流戻しで削除済（下流実験を残さない）。Docker イメージ `imagebind-bench` と checkpoint（`~/.cache/imagebind/imagebind_huge.pth`）はキャッシュ温存。
- **2026-06-15 に判明: Ollama は画像 embed 不可**。専用 embed モデルは `images` を無視、vision 生成モデルは `/api/embed` が **501 Not Implemented**。＝画像/横断 embedding は Ollama の外（＝上記 ImageBind サービス）で回す。
- **将来**: Ollama 等が横断 embed に対応したら横断 backend を差し替えるだけ（窓口・呼ぶ側は無変更）。

## 横断 embedding の設計（dual-vector・実装済 2026-06-16）

OFF 既定で実装済み（既存挙動ゼロ変更・全テスト緑）。細かいツマミは §4。

### オン/オフできる（デュアルの副産物）

横断は **additive ＝タダでオン/オフできる**。nomic 側（`OllamaEmbedClient`）は常に効くので、ImageBind 層を切れば**そのまま今の nomic だけ挙動**に戻る。`settings.crossmodal.enabled=false`（または host 未設定）で横断 embedder が `NullXmodalEmbedder`（`enabled=false`）になり、符号化は横断ベクトルを書かず recall は横断チャンネルを引かない＝degrade パスがそのままオフスイッチ。**禁止条項も特別分岐も要らない**。知覚エピソードは text ラベル（nomic）だけ持ち、知覚類似での想起ができないだけ。

### ① 2 本の embedder（窓口）

統一ファサードは作らず、**既存の nomic はそのまま据え置き＋横断だけ足す**（既存を触らない）。
- **text → `OllamaEmbedClient.embed`（768・nomic・既存・常時）**。
- **横断 → `XmodalEmbedder`（`src/embedding/xmodal.ts`・1024・ImageBind）**: `embed(input) -> number[1024] | null`。`input = {kind:"text"|"image"|"audio", ...}`。`ImageBindClient`（HTTP）/ `NullXmodalEmbedder`（OFF）/ `createXmodalEmbedder(settings.crossmodal)` factory。どんな失敗でも throw せず null＝degrade。

### ② ImageBind HTTP サービス（`scripts/imagebind-service/`）

- Python・Docker 常駐。`POST /embed {modality:"text"|"vision"|"audio", data:str}` → `{vector:[...1024]}`、`GET /health`。出力は **L2 正規化**して返す（横断グラデーション閾値の regime を [0,2] に揃える）。起動ロード 21s は常駐で1回。
- bot は `settings.crossmodal.host` を叩くだけ（Ollama と同じ作法）。落ちてたら null＝degrade。
- ピン（実証済・memory `project-crossmodal-embedding`）: py3.9 / torch2.1.2 / torchvision0.16.2 / numpy<2 / timm0.9.16。イメージ `imagebind-service`・checkpoint `~/.cache/imagebind/` 温存済（E2E 検証済）。

### ③ 横断ベクトルは別テーブル（`episodes_xmodal`）

- `episodes` に nullable vector を足すと固定長 list 列の地雷を踏むので、**横断ベクトルは別テーブル `episodes_xmodal`(id, vector1024)**（`src/memory/xmodal-lancedb.ts`）。`memo_index` と同じ「別テーブル（補助の記憶）」流儀。
- **移行不要**: 既存 text-only エピソードは横断テーブルに行を持たない。横断ベクトルは**これから入る知覚エピソード（画像/音）にだけ**付く。
- 忘却（forget）は episodes の softDelete と一緒に `episodes_xmodal` 行も `remove`。

### ④ recall の2空間マージ（設計の山場＝RRF で逃げる）

- nomic 距離（768）と ImageBind 距離（1024）は**物差しが違う**＝生スコアを足すのは脆い。
- **Reciprocal Rank Fusion**（`src/recall/fuse.ts`）: 各チャンネルで順位を取り `score = Σ 1/(k+rank)` で融合。スコア正規化が要らず異種 retriever 融合の定石。
- チャンネル（最大3本）: (a) nomic: text recall query × 全エピソード（既存 `episodes.recall`）。(b) 横断・テキストクエリ: `XmodalEmbedder.embed({kind:"text",...})` × `episodes_xmodal`＝**テキスト→画像/音の横断**。(c) 横断・画像クエリ: いまの `imageFeed[0]` を embed × `episodes_xmodal`＝**画像→画像の類似想起**。各々を RRF の別チャンネルにし、id で dedup、横断のみのヒットは `episodes.getByTurnIds` で本文をハイドレート（複数横断クエリに出たら近い方の距離を採る）。
  - ※(c) は**連想想起**（見た目が近い過去が浮かぶ・背景チャンネルに滲む）であって **identity の同定（recognition）ではない**。ImageBind はシーンレベルで identity を捉えない。「これはポチ」の同定は別 faculty（下「recognition」節）。
- テキストクエリが無くても（`buildRecallQuery`=null）、いまの景色（imageFeed）があれば**画像チャンネルだけで（連想）想起する**。その場合 nomic/意味記憶/memoIndex はスキップ。
- **想起グラデーション**: ヒットに `space`("text"|"xmodal") を付け、`classifyRecallHits` が空間ごとに閾値を出し分ける（横断は `DEFAULT_XMODAL_RECALL_DISTANCE_THRESHOLDS`・`settings.crossmodal.recallDistance` で上書き）。閾値の実値は §4 のツマミ。

### ⑤ 知覚の入口

- 符号化時（内省→LanceDB 書き込み・`turn.ts` `persistXmodalVector`）、その turn に知覚チャンネル（`imageFeed`/`audioFeed`）があれば**生入力を `XmodalEmbedder.embed`** して `episodes_xmodal` へ（画像優先）。text ラベルは常に nomic。best-effort（失敗しても本体は残る）。
- 想起時は④の通り imageFeed を画像クエリにして引く（画像→画像の類似想起・実装済）。※identity 同定ではない（下「recognition」節）。
- `audioFeed` チャンネル＋trigger の `audio?` は配線済みだが**音源 producer は未配線**（マイク/omni 音声入力が来たら繋ぐ）。音声の想起クエリ化（音→音）は producer が来てから。

### ⑥ 設定（`config/settings.json`）

`crossmodal = { enabled, host, timeoutMs, recallDistance? }`。未設定で OFF。量子化（fp16/int8）はサービス側で持つ想定（bot 設定には未追加）。

### ⑦ degrade

サービス落ち/オフ → 横断 embedder が null → 横断ベクトルを書かない・横断チャンネルを引かない。**nomic 側で想起は完全に生存**（systematic 起動の degrade と同じ・壊れない）。

## recognition（同一性の認識・未実装の faculty）

横断を入れて見えてきた、**recall とは別種の記憶アクセス**。2026-06-16 に整理。**まだ作っていない**（バグでなく未実装）。

### recall（想起）と recognition（認識）は別物

- **想起（recall）**＝手がかりから関連記憶が**ふわっと浮かぶ**。連想・salience重み・曖昧。**なぜ浮かんだかは知らなくていい**（背景recall は link-less が仕様。画像cueでも同じ＝上 §④(c)）。これは**正しい・実装済・据え置き**。
- **認識（recognition）**＝知覚したものを「**これはポチ／これはHAL**」と**即・確定的に同定**する。自動・明確・salience無関係。**知覚側の機能**で、背景recall とは別レーン。これが**無い**。

混同しないこと（この faculty を背景recall の importance/グラデーションで作ろうとするのは筋違い。確定的な同定に salience も曖昧化も要らない）。

### なぜ横断（ImageBind）では作れないか

ImageBind は**シーン/内容レベル**の汎用埋め込み。**identity を捉えない**（同一人物の別写真が遠く・別人の似構図が近くなりうる）。同定には **identity特化の埋め込み**が要る:
- **顔** → ArcFace / InsightFace（512d・CPU可・onnxruntime-node で in-process）
- **声** → ECAPA-TDNN / SpeechBrain / pyannote（~192d・CPU可）

人間も顔認識（紡錘状回）・声認識は一般知覚と別の専用機構＝**専用 recognizer を持つのが faithful**。＝横断の「1空間」とは別に、**顔空間・声空間という専用空間が増える**（embedding backend を足す。差し替え枠に乗る）。

### 要るもの（entity 層＝重い本丸）

1. **identity特化 embedding backend**（顔・声）。
2. **entity/exemplar ストア**：名前付き同一物 ↔ モダリティ別の見本埋め込み（複数）。
3. **命名・登録フロー**：会話で与えられた名前を今の顔/声の埋め込みに束ねる（entity 形成）。
4. **recognition module（preprocess 段）**：知覚チャンネルを読み、identity-model で embed → 最近傍 exemplar → 近ければ「これはX」を**専用 recognition チャンネル**へクリアに出す（背景の「参考のみ」枠ではなく、言語野が直接読む確定情報）。

### 育て方＝意味記憶と同型（シード＋夢）

- **シード先行**（推し）：世帯の既知メンバー（人・ペット）の顔/声＋名前を一度登録（`identity-seed` 的・`semantic-seed.json` と同じ思想）。家庭用エージェントは初日から確実に「お前を知ってる」＝期待挙動。かつ**身近な人の同一性を誤学習する作話を回避**（外界 grounded の歯止め）。
- **夢で拡張**：繰り返し会う人/物を、**ground が強い時だけ**（本人が名乗った時にその顔/声が支配的、等の高いバー）束ねて覚える。「身近な人は最初から知っている／新しい人は会ううちに覚える」＝人間そのもの。
- シードは哲学的妥協でなく「生得的な身近さ」に当たる。

### 能動 recall の画像/音声cue（旧 ③）は recognition に吸収＝単独では作らない

「これ覚えてる？〔画像/音〕」で意図的に引く案（memory actor に知覚cue）は、**recognition（同定）＋entity紐づき想起が上位互換**。横断の見た目類似で能動検索する別レーンは、同定が下手で本命になれない＝**作らない**。「見て覚えてるを引く」は recognition で解く。

## 4. あとで決めること（設計が固まってから・実装時のツマミ）

設計に混ぜない。懸念としては正しいので記録だけする。

- **フレームレートの値**（1個の細かい時計の速さ、各モジュールの間引き幅）。
- **遅いモジュールの読み直し**: 出来た時に過ぎたフレームを読み直して「まだ言う?」を判断するか。毎回やる（安全・重い）か、変化が一定以上の時だけ（軽い・「一定以上」を何で測るか要定義）か。今は「遅れ許容＋フレームレート調整」で先送り。
- **フレームの寿命**: 板にどれだけ残すか（古いフレームの落とし方）。長期に残すものは内省が LanceDB に符号化。
- ~~**活性化のコスト**~~ → §1.6「起動の段（none / systematic / llm）」で解決。cheap なモジュールは none/systematic で LLM を呼ばない。残るのは llm 段のモジュールを「入力が変わった時だけ再判断」するか等の実装ツマミだけ。
- **とっさの一言を内省がエピソードに含めるか**（作業記憶＝乗せる／長期＝残さない、までは合意。境界の明文化は実装時）。
- `LlmClient`/Ollama の streaming 対応と omni の運用（VRAM・`ollamaMaxConcurrency` の同時実行制限との兼ね合い）。
