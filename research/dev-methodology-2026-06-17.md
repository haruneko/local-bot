# 上流から下流まで整合を保ち続ける――local-bot の開発手法

> 本稿は、ローカル LLM（Ollama）で動く自宅向け対話エージェント「エバ」を実装するリポジトリ `local-bot`（2026-06-06 の `initial commit` 〜 2026-06-17 時点で 69 コミット）の一次記録――git 履歴、`docs/` 配下の設計文書、Claude Code のセッション生ログ（`~/.claude/projects/.../*.jsonl`、最古は 2026-06-07）――から、**実際にどう開発されてきたか**を再構成したものである。記録から読み取れる中心的な手法は、コンセプト（WHY）から SPEC・決定ログ・コード・運用規約（HOW）までを一本の鎖として扱い、**鎖のどこかがズレたら都度揃え直す反復**だった。以下、すべての主張に出典（コミットハッシュ・文書名と節・日付・セッション引用）を添える。推測が混じる箇所は「記録上は」と限定する。

## 1. 何を作っているか

`docs/CONCEPT.md`（初版 v0.4、現 v0.5）の言葉では、「実行・言語化・内省」を構造的に分離し、小さいモデルでも破綻しにくくする最小認知アーキテクチャである。分離脳の左脳モデルに倣い、言語化と内省は行動の「原因」ではなく「結果」として後付けで生成される。単一の意思決定者（ジャッジ）を置かず、各エージェントが同一の観測事実（`TurnContext`）から自律的に判断して動く。本稿はこの中身そのものより、**それがどう作られてきたか**を扱う。

## 2. doc-first の立ち上がり

特徴的なのは、コードと設計文書がほぼ同時に存在していた点である。`git show --stat 4ec43a9`（initial commit, 2026-06-06）の時点で、`docs/CONCEPT.md`（170 行）・`docs/SPEC.md`（237 行）・`docs/DECISIONS.md`（123 行）・`docs/ACTION-DESIGN.md`（171 行）が `src/` 一式と同じコミットに含まれている。SPEC は冒頭から「CONCEPT.md v0.4 と DECISIONS.md に基づく実装契約。**テストは本書の MUST 節を根拠に書く**」と宣言しており、最初から「思想（CONCEPT）→契約（SPEC）→判断ログ（DECISIONS）」という役割分担が敷かれていた。

この役割分担は記録から明確に読み取れる。

- **CONCEPT.md** = WHY。設計原理（分離脳・単一意思決定者を置かない）。
- **SPEC.md** = 実装契約。MUST 節を持ち、テストの根拠になる。
- **DECISIONS.md** = 決定ログ。冒頭に「CONCEPT.md の思想は変えず、実装判断だけをここに固定する。これは**ログ**なので、過去の決定（後に置き換わったものを含む）が時系列で積まれている」とある。
- **CLAUDE.md** = AI エージェント（Claude）への運用規約。

CLAUDE.md は最初期の開発会話で生まれている。記録上もっとも古いセッション（`5b77ffae...`, 2026-06-07T01:36Z）の人間（開発者 HAL）の最初の発話は「docs 下と README.md 読んで CLAUDE.md 作ってくれる？」だった。つまり**設計文書が先に在り、それを読ませて AI 向けの運用規約を作る**という順序が立ち上げ時から取られている。

## 3. 上流↔下流の整合プロセス（本稿の主題）

開発の駆動力は、コンセプト→SPEC→DECISIONS→コード→docs/CLAUDE.md という鎖を**一貫させ続ける反復**にある。記録には二つの面で現れる。

### 3.1 上流に照らして下流を決める

2 番目に古いセッション（`69f80609...`, 2026-06-07）には、判断の基準を毎回コンセプトに引き戻す動きが繰り返し出てくる。会話履歴の圧縮アルゴリズムについて HAL はこう言う:

> 「あー、圧縮に哲学がないのか。この圧縮ってとりあえずなんでもいいからぎゅっと圧縮しちゃうってことでしょ？？**コンセプトと照らし合わせると**どういう圧縮にするのがいいかな？」

エピソード記憶の要約についても:

> 「ここがぼんやりするのは**コンセプトとして正しい気がする**けど一方で完全に想起した内容がボケまくっても会話が死ぬ気がしていて雑に全部ぶっ潰して要約していいんだっけ？」

そして同セッションの終盤、自他境界（誰が言ったかの取り違え）が直らない局面で、人間は対症療法を止めて上流に戻る判断をする:

> 「んー、そうなると**アーキテクチャ全部見直しじゃない？？新しいプラン立てて始めたい**けどどうしたらいい？？」

この「下流の不具合を、上流（アーキテクチャ）の見直しとして扱う」動きが、後の大きな構造変更（`fa18129` Change architecture to expand actions, `af2de6f` No more Judge needed, `2ca0407` flatten all actors）につながっていく。記録上、ジャッジ廃止と actor のフラット化は思いつきではなく、上流の原理（単一意思決定者を置かない）に実装を合わせ直す作業として現れている。DECISIONS.md の「エージェント設計（v0.6→v0.7）」節は、この経緯を「v0.6 の『ジャッジ廃止』から一歩進め、エージェント内部の『束ね』も除去する」と時系列で残している。

### 3.2 下流が動いたら上流（docs）を揃え直す

逆方向、つまり**実装が進んだあとに docs を実装へ合わせる**専用のコミットが繰り返し打たれている。コミットメッセージは接頭辞で種類が分類されており（「整理:」「想起:」「記憶:」「集中:」「修正:」「docs:」「洗い直し:」など）、整合作業はその一群として識別できる。代表例:

- **`c03cce5`（2026-06-14）** "docs を実装に整合（記憶faculty B'・importance採点位置・synthesize漏れ）"。`CLAUDE.md` / `ACTION-DESIGN.md` / `CONCEPT.md` / `DECISIONS.md` / `SPEC.md` の 5 文書を**1 コミットでまとめて**触り、「SPEC actor 表に欠けていた synthesize を追加」「importance 採点を『内省』→『内心更新(affect)と同じ呼び出し』に表記統一」など、実装の registry と文書の不一致を潰している。
- **`c9e0592`（2026-06-15）** "docs を実装に整合（research=要約のみ・LLM同時実行リミッタ・夢の grounding）"。本文に「**c03cce5 以降の実装変更が docs に未反映だった点を揃える**」とあり、整合が一度きりでなく**継続的に再実施される**ことが明示されている。CONCEPT の「意味記憶の蒸留は将来実装」という記述を「バッチ実装済」に訂正する、といった上流文書の事実合わせも含む。
- **`834b5d8`（2026-06-15）** "洗い直し: legacy(searxng/docker)撤去＋活性化原則を起動の段と整合"。死んだ依存（searxng/Docker、Tavily API へ置換済みで参照ゼロ）を消すと同時に、「旧『活性化判断は必ず LLM』は非同期化と矛盾し recall/distill で破綻していた」として、CONCEPT・DECISIONS×2・CLAUDE を**横断で**書き換え、原則そのものを「起動が"状況の判断"か"客観条件"か」で段分けする形に整え直している。
- **`5c1e7bf`（2026-06-16）** "整理: 想起ドキュメントを vague 廃止に整合"。直前のコード変更 `5cb220e`（vague 段の廃止）に合わせ、文書側の記述を揃える。

記録から読み取れるのは、整合が**後付けの掃除**ではなく**工程として組み込まれている**ことである。実装（下流）が動くたびに、それを上流の文書へ反映する整合コミットが対で打たれる。

## 4. 人と AI の分担

セッションログと git 履歴を突き合わせると、役割は概ね次のように分かれている。記録上、人（HAL）は上流の判断と整合チェックを主導し、AI（Claude）は実装・検証・記録を担っている。

- 立ち上げ時、HAL は「今後実装する際は**必ず私に方針相談してから実施してほしい**。OK？」（`69f80609`, 2026-06-07）と分担を明示している。実装着手の前に方針合意を置く運用は、本リポジトリの `/impl`（実装着手チェックリスト）にも残る。
- 方針が固まると、HAL は記録への落とし込みを AI に指示する。「オーケー、じゃあ docs を読んだうえで**設計や決定を docs に起こしてからテスト駆動で実装して**くださいな」（セッション `992caf7a`）。これは「合意→DECISIONS/SPEC へ記録→TDD で実装」という順序を一文で表している。
- 議論の確定を文書化につなげる意識も繰り返し出る。アクション側がエピソード記憶を作る変更について HAL は「後々破綻しないかな…一応論戦してから確定させたい。**この辺の議論固まったら DECISIONS.md には記載したいね**」（セッション `d3ac37ac`）と述べている。

git 側の裏付けとして、69 コミット中 49 コミットが `Co-Authored-By: Claude Opus 4.8` を持つ（最初期の `Fix`/`fixed` 系を除くほぼ全期間）。整合コミットの詳細な本文（変更点を箇条書きで、なぜ揃えるかまで書く）も、AI が実装・検証・記録の実行役を担っている形と整合する。

## 5. 規律――記録に現れる範囲で

手法を支えているガードレールも、文書とコミットに痕跡がある。

**MUST を根拠にした TDD。** SPEC.md は冒頭で「テストは本書の MUST 節を根拠に書く」と定め、CLAUDE.md の「テスト方針」も「テストは SPEC の MUST 節を根拠に書く（TDD）。LLM 統合テストは持たず、`src/llm/fake.ts` / `src/mcp/fake.ts` のフェイクを使う」と繰り返す。テストが**仕様の MUST に紐づく**ため、契約とテストとコードが同じ根に揃う。

**理由つきの決定ログ。** DECISIONS.md は決定を「動機 / 却下 / 採用」の粒度で残す。たとえば記憶系アクター統合（B'・2026-06-14）の節は、**動機**（recall/forget/memo が独立並列で縄張り衝突し「記憶から消して」に memo が誤爆した）、**却下**（「記憶統括くん」＝記憶ドメインのジャッジは「CONCEPT の核（単一の意思決定者を置かない・分離脳の左脳モデル）に反する」）、**採用**（HAL の決め手「記憶≠記録。記録は能動的にやる以上それが正本だから別物として束ねる」を引用）の三点を残している。「却下」を**理由つきで残す**ことで、なぜその道を選ばなかったかが後から辿れる。同様の「却下」記載は abstract mood scalar、agenda フィールド、記憶統括くん等、複数の節に見られる。

**やってはいけないこと（ガードレール）。** CLAUDE.md は「やってはいけないこと（DECISIONS.md より）」を節として持つ。「キーワードマッチでの直行ルーティング」「ロールごとに想起・会話ログの入れ方を変えること」「メモ本文を LLM で要約すること（劣化するから）」「フィルタを理由に元データ（LanceDB・state.json）を削除・変更すること」など、**上流の原理を破る具体的なアンチパターン**を列挙している。これは AI に毎回コンテキストとして渡る規約であり、整合プロセスで確定した原則を**実装時に踏み外さないための歯止め**として機能している（記録上、これらは DECISIONS の決定エントリと対応する）。

## 6. 結び

local-bot の開発記録から見えるのは、派手な機能追加よりも、**上流（コンセプト）と下流（コード・文書）のズレを検知して都度揃え直す**地道な反復である。人が「コンセプトと照らし合わせる」「アーキテクチャごと見直す」と上流に引き戻し、議論が固まれば DECISIONS/SPEC に記録させ、実装が動けば「docs を実装に整合」コミットで下流から上流へ反映を返す。AI はその実装・検証・記録を実行する。

なお、この「意図（上流）を先に固め、実装をそれへ収束させ、ズレたら戻って整える」反復は、近年の上流駆動なアジャイル/AI 開発手法（例として AWS の AI-DLC）の発想と、独立に似ている点が少しある。ただし local-bot の記録は特定の方法論を導入した形跡を示しておらず、本稿はあくまで一次記録に現れた実際の進め方を記述したものである。
