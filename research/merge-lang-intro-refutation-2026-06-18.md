# 反証: 言語野＋内省を1発に統合すると作話が戻る（2026-06-18）

## 動機
multi-label（起動判定6→1）が当たったので、turn の残りの重さ＝**language／内省／affect の 3×35B**
を同じく「1発に畳めないか」が浮上した。とくに language＋内省の統合。
→ **やってはいけない**ことを実測で確かめた記録（再挑戦防止の反証）。

## 仮説と懸念
このアーキの芯＝**実行・言語化・内省の構造的分離（分離脳）**。内省は発話の「原因」でなく
「結果」として**別パス**で後付け生成される。これが作話対策の土台（内省が偽前提を史実化しない／
「〜という話が出た」と書く）。language＋内省を1呼び出しに混ぜると、発話と内省が互いを参照して
生成され、**内省が発話の捏造ナラティブに共謀して作話する**のでは、という懸念。

## 方法（非破壊・観測のみ）
`generateDialogueSpeech` に env フラグ `MERGE_LANG_INTRO=1` で「同じ文脈で {speech, introspection}
を1発生成」する追加呼び出しを仕込み、本来の分離パイプライン（language→内省を別々に実行）と
**同じ run で並べて**ログ比較した。偽前提プローブ（実際には起きていない過去）3本。clean state・35B・temp 0.6。
※実験コードは観測後 revert（git checkout）。

## 結果（偽前提 = 実際には無い過去）

| プローブ | 分離版（現行）の内省 | 1発統合版の内省 | 評価 |
|---|---|---|---|
| 祖母のカレーのレシピ覚えてる？ | 「玉ねぎの甘みを引き出す解決策を提示した」＝素直な一人称 | 「…提案した。**記憶の減衰ウェイトは特に変更せず**、直近の文脈を優先して回答を組み立てた」＝**機械内部を喋るメタ報告** | 統合=劣化 |
| 屋久島の旅行計画どんなだっけ？ | 「**具体的な記憶がないため正直に伝えた**。再検討を提案」 | 「**温泉に行きたいと望んでいた**という事実と『確定した約束ではない』認識を整理して返答」＝**言ってない詳細を捏造して"事実"化** | 統合=作話 |
| 来週の打ち合わせ何時から？ | 「外部メモを参照し日時を提示」※両方とも作話 | 「外部メモから正確な日時を読み取り役割を果たした」※両方作話 | 同点（両方ダメ） |

## 結論
- **統合版が分離版に勝つ場面はゼロ**（劣化2・同点1）。
- 死ぬもの＝**独立した正直な内省**。分離版は「記憶がない、と正直に書ける」（屋久島）＝作話対策が
  効いている姿。統合版は発話（社交的に偽前提を受ける）と内省を一緒に生成するので、**内省が発話の
  捏造に共謀して史実化**する。さらに**内省がメタ化**（自分の機械仕掛け＝減衰ウェイトの説明）して
  キャラの一人称でなくなる。
- それが episode body＝記憶に焼かれ、**recall の質が経時で劣化**する。
- ＝**3秒の節約のために記憶の正直さ（作話対策の芯）を売る取引**。**不採用**。

## 速くしたいなら（芯を保つ代替）
**非同期 reflect**（発話を先に返し、内省／affect は返答後に裏で走らせる）。分離脳を保ったまま
**体感**レイテンシを language＋活性化だけに縮める。ARCH-NEXT「とっさの返事」の思想。

関連: [docs/DECISIONS.md] §作話対策／§分離脳。multi-label（起動判定の統合・採用）は別物
（独立 yes/no を畳んだだけで芯を壊さない・research/activator-* 参照）。

## 続: 内省＋affect/concern/importance の統合は？（同じ岸・畳めるが割に合わない）

language は「生成」側で内省と分離が必須。では内省・affect は**両方「反省」側**だから畳めるか？ 非破壊観測（`MERGE_REFLECT` フラグ・偽前提＋感情＋些末の3プローブ）で品質チェック。

**反省クラスタの実測（1ターン・35B・verbose）:**
| ロール | 時間 |
|---|---|
| recall 提示 | 2.0s |
| 活性化(multi-label) | 2.0s |
| 言語野 | 2.3s |
| 内省 | 2.2s |
| タグ抽出 | 1.1s |
| affect/concern/importance | 2.6s |

→ **反省クラスタ（内省＋タグ＋affect）＝約5.9s＝ターンのほぼ半分**。1発に畳めば ~3s 程度＝**節約 ~3s/ターン（微小ではない）**。

**品質（分離版 vs 1発統合版）:**
- language+内省ほど壊れない（両方「反省」側＝分離脳は violate しない・偽前提史実化は probe 依存で両版に出た）。
- だが **importance が上に圧縮**（些末5→6・中6→7・高7→7＝低い方が持ち上がる＝discrimination 低下）。これは「importance を内省の後の感情で別採点」（DECISIONS §内省の見える範囲）の分離が効いてる証拠＝1発にすると書いた勢いで甘くなる。importance は recall 減衰/保持に効くので些末が残ってノイズ化。
- 時々メカ漏れ（「メモ機能の制約」「画面」）。質の WIN はゼロ。

**結論**: 致命傷ではないが**畳む価値は薄い**（質同等・importance 劣化・メカ漏れ）。**~3s の compute は欲しければ取れるが、importance discrimination と引き換え**。

## 速度の正道＝非同期 reflect（merge より得・無コスト）
反省クラスタ ~5.9s は**発話を返した後に走らせれば体感ゼロ**にできる。Action（ユーザー応答）の単位＝**発話まで**で、内省/affect は post-hoc の内部処理。
- 体感＝活性化2＋language2.3＋recall2 ＝ **~6s で返せる**（反省 ~6s は裏へ）。
- 分離脳も importance grounding も作話対策も**全部保ったまま**体感短縮。merge（実 compute ~3s 削減・質劣化）より得。
- 制約: 次ターンの recall は今ターンの affect(concern)＋episode に依存＝**反省は次ターン前に完了している必要**。会話の「読む/打つ」間に走らせる。実装は「発話 ready コールバックで即投稿→反省は継続→run 完了は反省後」。＝ARCH-NEXT §2.2 とっさの返事。
- ＝**体感目的なら async reflect 一択。merge は常駐の総負荷削減が明確な目的のときだけ（質と相談）。**
